# Next.js SaaS Render Netlify Starter

Generated by `stripe build` as the first supported vertical slice.

## What this starter gives you

- Next.js App Router app
- Bare-bones Stripe Checkout subscription flow for $10.00 / month
- Scripted Stripe product and recurring price setup with `npm run setup:stripe`
- Optional Clerk auth wiring with checkout protection
- Optional Render Postgres wiring with starter `users` and `subscriptions` tables
- Optional Netlify hosting metadata
- Optional Mixpanel analytics wiring with autocaptured pageviews and conversion events


## Verify the starter

Open the landing page and use it as the setup summary for the generated app.

- `Subscribe` opens the starter subscription flow
- `View Database` opens the same starter database summary modal used in the integration panel
- `View sign in` verifies the generated Clerk entry points, then sends signed-in users to `/dashboard`
- `View Database` shows the starter Render Postgres connection state plus row counts for `users` and `subscriptions`
- `npm run deploy` uses the pulled Netlify credentials to publish the app


## Local development

1. Install dependencies:
   `npm install`
1. Create the starter Stripe product and price:
   `npm run setup:stripe`
1. Verify that `STRIPE_PRODUCT_ID`, `STRIPE_PRICE_ID`, and `STRIPE_SECRET_KEY` are present in `.env.local`.
1. When Render Postgres is configured, the starter automatically creates its `users` and `subscriptions` tables the first time a database-backed route runs.
1. When Clerk is enabled, direct sign-ins and sign-ups flow through `/auth/sync`, and signed-in checkouts also upsert the app user before creating the Stripe Checkout Session.
1. To enable Stripe webhook syncing after the app has a public URL, create a Stripe webhook endpoint for `https://your-app.example.com/api/webhooks/stripe` and add its signing secret to `STRIPE_WEBHOOK_SECRET` in `.env.local` or your deployment environment.

1. `npm run deploy` publishes to Netlify when these pulled env vars are available: `NETLIFY_NETLIFY_AUTH_TOKEN`, `NETLIFY_NETLIFY_SITE_ID`, `NETLIFY_NETLIFY_SITE_NAME`, `NETLIFY_NETLIFY_SITE_URL`.

1. If the generator did not finish provisioning your add-ons, run:
   - `stripe plugin install projects`
   - `stripe projects init`
   - `stripe projects add netlify/project`
   - `stripe projects add render/postgres`
   - `stripe projects add clerk/auth`
   - `stripe projects add mixpanel/analytics`
   - `stripe projects env --pull`
1. Start the app:
   `npm run dev`

## Routes

- `/` landing page
- `/dashboard` signed-in starter workspace

- `/checkout` starter subscription checkout flow
- `/success` checkout success return page
- `/cancel` checkout cancel return page
- `/api/health` runtime configuration summary
- `/auth/sync` best-effort Clerk-to-database sync route
- `/api/billing-portal` creates Stripe customer portal sessions for signed-in subscribed users
- `/api/db` database summary route with starter table counts when Render Postgres is provisioned
- `/api/webhooks/stripe` Stripe webhook endpoint for initial subscription linking once `STRIPE_WEBHOOK_SECRET` is configured

The prompt tells the agent to read `AGENTS.md`, inspect the starter, ask one discovery question at a time when the brief is still fuzzy, confirm the product name with the user, work in phases with the landing page first, remove starter status UI from the product experience, and only then implement without breaking the existing Stripe, auth, database, and deployment wiring. It also points the agent at a frontend design reference to avoid generic AI-starter styling.
